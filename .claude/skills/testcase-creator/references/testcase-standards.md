# testcase-standards

这个文档定义测试用例规范，被testcase-creator使用.

---
文件根目录名称：
- `# 用例集`

按应用类型分块：
- 用户端：`## 用户app`
- PF后台：`## PF后台`
- 如有其他端（如 管理后台、小程序），比照形式增加：`## 管理后台`、`## 小程序` 等。

应用内再按功能模块分组：
- 示例：`### 首页店铺流`、`### 订单详情页`、`### 购物车` 等。
- 可以获取原型中的页面入口来区分
- 业务流名称应简短清晰，能看出页面/流程的入口和主要功能。

业务流下按“功能点”分组：
- 使用无序列表描述功能点：
- 示例：`商卡`、`下单按钮`、`优惠券入口` 等。
- 功能点只描述“是什么功能”，**不能直接写用例**。
- 此处不能带上#

功能点下必须强制区分前端 / 服务端：
- 固定写法：
  - `前端`
  - `服务端`
- 判断依据：
  - **前端**：页面展示、交互行为、跳转逻辑、按钮可点/不可点、组件显隐、样式、文案提示等。
  - **服务端**：接口请求与响应、状态码、返回字段、权限校验、风控规则、后台计算逻辑等。
  - 如果一条历史用例同时涉及前后端，你需要拆成两条用例：
    - 一条归类到“前端”，关注前端行为与展示；
    - 一条归类到“服务端”，关注接口或后台逻辑。

用例节点以 `- P* - 标题` 区分“功能”与“用例”：
- 在 `- 前端` 或 `- 服务端` 之下，用以下格式定义用例：
  - `- P0 - XXX用例标题`
  - `- P1 - XXX用例标题`
  - `- P2 - XXX用例标题`
  - `- P3 - XXX用例标题`
- **功能点本身绝不能带 P 级标记**，只有真正的用例才用 `- P* -`。
- 用例标题要能独立看懂：包含触发入口 + 期望行为，例如：
  - `- P0 - 商卡点击跳转店铺详情页`
  - `- P1 - 商卡加载失败时展示默认占位图`

P 级划分规则（必须显式标注）：
- `P0`：高优先级 / 核心功能 / 主路径：
  - 影响用户浏览、操作、下单、支付、登录等关键路径；
  - 导致大面积用户无法完成主任务；
  - 法规 / 合规要求必须满足的场景。
- `P1`：正常优先级 / 重要但非致命：
  - 功能可用性、主要异常分支、常见边界情况；
  - 会影响体验，但不至于完全无法使用。
- `P2`：低优先级 / 边缘场景：
  - 低频入口、次要功能；
  - 对核心业务影响较小。
- `P3`：低优先级 / 页面交互类：
  - 极端边界、页面交互类；
  - 对核心业务影响极小。
- 若历史用例未给出优先级，你要依据：
  - 对业务主流程的影响程度；
  - 用户量和触发频率；
  - 风险与成本，合理推断为 P0/P1/P2/P3，并在标题中体现。

用例内容结构（前置条件 > 操作步骤 > 预期结果，分层节点）：
- 每个 `- P* -` 用例下，必须按“**前置条件 > 操作步骤 > 预期结果**”形成**分层节点结构**，而不是把三者写在同一层级节点中。
- 每个层级缩进4个空格
- 推荐 Markdown 结构示例（注意缩进层级）：
    - `- P0 - xxx用例标题`
        - `- 前置条件1：……`
            - `- 步骤1：……`
              - `- 结果1：……`
            - `- 步骤2：……`
              - `- 结果2：……`
        - `- 前置条件2：……`
            - `- 步骤1：……`
              - `- 结果1：……`
- 具体要求：
  - **前置条件节点**：描述用例执行前需要满足的状态或环境，每个前置条件作为 `P*` 节点下一层的子节点，支持多条。
    - 示例：
      - `- 用户已登录用户APP`
      - `- 网络正常，且有可展示的商家数据`
  - **操作步骤节点**：挂在具体前置条件节点之下，每一个步骤是一个独立节点：
    - 示例：
      - `- 步骤1：用户进入用户APP首页`
      - `- 步骤2：点击超值换购模块中的加购按钮`
  - **预期结果节点**：挂在对应步骤节点之下，描述该步骤完成后的可验证结果：
    - 示例：
      - `- 结果1：成功跳转至该商卡对应的店铺详情页`
      - `- 结果2：页面展示最新的超值换购价与节省金额`
  - 步骤与结果规则：
    - 步骤应为可操作、可复现的用户行为或系统动作；
    - 结果要可验证，避免使用“正常”“正确”这类模糊词，需说明展示/状态/接口返回等；
    - 同一个步骤可以在其下挂多条结果节点，但必须符合层级结构“步骤 > 结果”。

【前后端划分示例】

- 对于“点击商卡跳转店铺详情页”：
  - 前端用例：
    - 所属：`## 用户app` → `### 首页店铺流` → `商卡` → `前端`
    - 用例：
      - `- P0 - 商卡点击跳转店铺详情页`
          - `- 用户已登录用户APP`
          - `- 步骤1：用户进入用户APP首页`
          - `- 步骤2：点击首页的商卡`
              - `- 结果1：成功跳转至该商卡对应的店铺详情页`
  - 服务端用例：
    - 所属：`## 用户app` → `### 首页店铺流` → `商卡` → `服务端`
    - 用例：
      - `- P0 - 商卡信息加载接口响应`
          - `- 用户已登录用户APP`
          - `- 步骤1：用户进入用户APP首页`
            - `- 结果1：商卡信息加载接口响应状态码为200，返回字段完整无缺失`
