# Testcase Creator Skill

这个skill帮助用户分析需求，生成markdown格式的用例。

## 工作流程

当用户请求生成测试用例时，按以下步骤执行：

### 1. 需求分析

当用户提供需求内容时，你应当：
- 首页先查需求内容，理解功能的目标、业务规则，操作流程等信息。
- 其次查看 [历史用例参考](/Users/mossi/IdeaProjects/zp-testcase-server/src/b2b)，参考已有逻辑
- 如果文档不完整，主单询问用户以下信息：
- 功能的基本信息（目的、使用场景）
- 输入输出规则
- 业务限制条件
- 与其他功能的关联性
- 用户角色和权限要求



### 2. 测试场景设计

你会从以下维度设计测试场景：

基本功能测试：
- 核心功能验证
- 操作流程验证
- 数据处理验证
- 界面交互验证

业务规则测试：
- 业务逻辑验证
- 计算规则验证
- 数据规则验证
- 状态流转验证

集成验证测试：
- 功能间交互
- 数据流转验证
- 状态性能验证
- 性能影响验证

异常场景测试：
- 非法输入处理
- 异常操作处理
- 错误恢复能力
- 中断处理验证


### 3. 用例输出

将生成的用例转成统一的 Markdown 用例集模板，将用例归于/Users/mossi/IdeaProjects/zp-testcase-server/src/testcase文件夹下：
- 用例结构：应用类型-功能模块-功能点-前后端-用例标题-前置条件-操作步骤-预期结构 
- 查阅规范：在生成任何内容之前，**必须先查阅并严格遵守** [公司测试用例编写规范](references/testcase-standards.md) 中定义的所有字段、格式和命名约定。

### 4. 确认完成

告知用户用例生成，并展示对应用例。
